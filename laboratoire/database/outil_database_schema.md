# 🗄️ Database Schema Management

> **Commande** : `git-ia-db`  
> **Statut** : 📝 Idée à explorer  
> **Priorité** : 🟣 Basse  
> **Effort estimé** : 50 heures

---

## 📋 Vue d'ensemble

`git-ia-db` automatise la gestion des schémas de base de données et des migrations.

---

## 🎯 Fonctionnalités principales

### 1. Génération de migrations
```bash
# Génère migration depuis diff de schémas
git-ia-db generate-migration --from old_schema.sql --to new_schema.sql --output migrations/001_add_users_table.sql

# Génère migration depuis modèles ORM
git-ia-db generate-migration --from-orm --orm sqlalchemy --output alembic/versions/xxx.py
```

**Exemple de migration générée** :
```sql
-- Migration: 001_add_users_table
-- Generated by git-ia-db

-- Up migration
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_users_email ON users(email);

-- Down migration
DROP INDEX idx_users_email;
DROP TABLE users;
```

---

### 2. Explication de migrations
```bash
# Explique une migration existante
git-ia-db explain-migration --file migrations/001_add_users_table.sql

# Sortie :
# 🤖 Analyse IA de la migration :
#
# Cette migration crée une nouvelle table 'users' avec :
# - Clé primaire auto-incrémentée (id)
# - Contrainte UNIQUE sur email (évite doublons)
# - Index sur email (optimise recherche par email)
# - Timestamp automatique à la création
#
# Impact estimé :
# - Temps d'exécution : < 1 seconde (table vide)
# - Risque : Faible
# - Rollback : Possible via down migration
```

---

### 3. Plan de rollback
```bash
# Génère plan de rollback pour une migration
git-ia-db rollback-plan --migration 001_add_users_table.sql

# Sortie :
# 📋 Plan de rollback :
#
# 1. Sauvegarder données (backup)
# 2. Exécuter down migration
# 3. Vérifier intégrité (foreign keys)
# 4. Tester l'application
#
# Commandes :
# pg_dump -t users > backup_users.sql
# psql -f migrations/001_add_users_table_down.sql
```

---

### 4. Validation de schéma
```bash
# Valide cohérence schéma DB vs modèles ORM
git-ia-db validate --orm sqlalchemy --database postgres://localhost/mydb

# Détecte :
# - Tables manquantes
# - Colonnes manquantes/en trop
# - Types incompatibles
# - Indexes manquants
```

**Exemple de sortie** :
```
🔍 Validation schéma vs modèles ORM

❌ 3 incohérences détectées :

1. Table 'orders' manquante en DB
   💡 Modèle ORM : Order (models.py:45)
   💡 Créer migration : git-ia-db generate-migration --add-table orders

2. Colonne 'users.phone' manquante en DB
   💡 Modèle ORM : User.phone (models.py:12)
   💡 Créer migration : ALTER TABLE users ADD COLUMN phone VARCHAR(20);

3. Type incompatible : users.age
   💡 DB : VARCHAR(10)
   💡 ORM : Integer
   💡 Corriger type : ALTER TABLE users ALTER COLUMN age TYPE INTEGER;
```

---

### 5. Optimisation de schéma
```bash
# Suggère optimisations
git-ia-db optimize --database postgres://localhost/mydb --analyze

# Détecte :
# - Indexes manquants (requêtes lentes)
# - Indexes inutilisés
# - Colonnes non normalisées
# - Tables sans clé primaire
```

## 🎓 Exemples d'utilisation

### Scénario 1 : Nouvelle fonctionnalité nécessite nouvelle table

```bash
# 1. Modifie modèle ORM
vim models.py  # Ajoute class Order

# 2. Génère migration automatiquement
git-ia-db generate-migration --from-orm --orm sqlalchemy

# ✅ Migration générée : migrations/002_add_orders_table.py

# 3. Explique la migration
git-ia-db explain-migration --file migrations/002_add_orders_table.py

# 4. Valide avant application
git-ia-db validate --migration 002_add_orders_table.py --dry-run

# 5. Applique migration
alembic upgrade head

# 6. Vérifie
git-ia-db validate --database postgres://localhost/mydb
```
